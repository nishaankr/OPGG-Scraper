# OP.GG Match History Scraper

A Python web scraper that extracts full match history from any OP.GG summoner profile and saves it to a CSV file.

---

## What It Extracts

| Field    | Example          |
| -------- | ---------------- |
| Result   | Victory / Defeat |
| Kills    | 13               |
| Deaths   | 5                |
| Assists  | 18               |
| Duration | 41m 06s          |
| Summoner | Afroyobro2-EUW   |
| Region   | EUW              |
| PUUID    | abc123...        |

This is what a single match card looks like on OP.GG — the scraper extracts Result, KDA, and Duration from each one:

![Match Card](Assets/match_card.jpg)

---

## Requirements

- Python 3.8+
- Google Chrome installed

```bash
pip install greenlet --only-binary=:all:
pip install playwright requests
python -m playwright install chromium
```

> **Windows users:** The `--only-binary=:all:` flag is required for `greenlet`. Without it, pip tries to compile from C++ source and fails on older Visual Studio versions.

---

## How To Use

```bash
python opgg_Extracter.py
```

Paste any OP.GG profile URL when prompted:

```
Paste your OP.GG profile URL: https://op.gg/lol/summoners/euw/Afroyobro2-EUW
```

The script will automatically load all matches by clicking "Show more", extract every match, and save to CSV.

## How It Was Built

Building this scraper went through three distinct approaches before landing on a working solution.

### Attempt 1 — requests + BeautifulSoup

The standard scraping approach failed immediately. OP.GG is built with Next.js — a plain HTTP request returns an empty HTML skeleton with no match data in it. All content is rendered by JavaScript in the browser after load, so BeautifulSoup sees nothing useful.

### Attempt 2 — RSC Payload Interception

Next.js apps stream data via React Server Components (RSC). Sending `RSC: 1` as a request header should return structured data directly — no browser needed. The response came back at only 307 characters: just a router navigation tree. OP.GG requires a dynamic session token generated by client-side JavaScript to return actual data. This approach was a dead end.

### Attempt 3 — Playwright (Final Approach)

The only reliable solution was to automate a real browser using **Playwright**, which launches Chromium, executes JavaScript, and reads the fully rendered DOM.

**Key challenges solved along the way:**

- **Windows install error** — `greenlet` failed to compile from C++ source. Fixed with `--only-binary=:all:`.
- **Bot detection** — Playwright sets `navigator.webdriver = true` by default, which OP.GG detects. Fixed by overriding the property with an injected script before page load.
- **Tailwind CSS selectors** — OP.GG uses utility class names like `flex gap-2` with no semantic identifiers. A DOM inspector script walked every text node to find the real element hierarchy.
- **KDA in sibling divs** — Result and duration were in one container; KDA was in a completely separate sibling div. Fixed by walking UP the DOM tree from each Victory/Defeat node until finding an ancestor containing both a KDA pattern and a duration pattern.
- **"Show more" button** — Playwright's `.click()` failed because the button was flagged as not visible. Fixed by firing a raw JavaScript `btn.click()` directly, bypassing Playwright's visibility checks entirely.
- **Wrong button** — Initial text matching used `"Show More"` (capital M) but the actual button reads `"Show more"`. This caused the script to click `"Show More Detail Games"` (a match expander) 20 times instead of loading new matches.
- **Duplicate rows** — The DOM tree walker visited nested elements multiple times for the same match. Fixed with a JavaScript `Set` deduplicating by `result+kda+duration` key.

## Notes

- Reads only publicly visible data, the same as opening OP.GG in a browser
- Speed is limited by OP.GG's server response time after each "Show more" click
- If OP.GG updates their page layout or button text, selectors may need updating
