# OP.GG Match History Scraper

A Python web scraper that extracts full match history data from any OP.GG summoner profile and saves it to a CSV file. Built with Playwright for browser automation and requests for PUUID extraction.

---

## What It Extracts

| Field    | Example          |
| -------- | ---------------- |
| Result   | Victory / Defeat |
| Kills    | 13               |
| Deaths   | 5                |
| Assists  | 18               |
| Duration | 41m 06s          |
| Summoner | Afroyobro2-EUW   |
| Region   | EUW              |
| PUUID    | abc123...        |

---

## Requirements

- Python 3.8+
- Google Chrome installed

Install dependencies:

```bash
pip install greenlet --only-binary=:all:
pip install playwright requests
python -m playwright install chromium
```

> **Note for Windows users:** The `--only-binary=:all:` flag is required for `greenlet` on Windows. Without it, pip tries to compile from C++ source and fails if your Visual Studio version is too old.

---

## How To Use

Run the script:

```bash
python rsc_extract.py
```

When prompted, paste any OP.GG summoner profile URL:

```
Paste your OP.GG profile URL: https://op.gg/lol/summoners/euw/Afroyobro2-EUW
```

The script will:

1. Validate the URL format
2. Fetch the summoner PUUID from the page HTML
3. Launch a headless Chromium browser
4. Load the full match history by clicking "Show more" automatically
5. Extract result, KDA, and duration from every match card
6. Print a summary table to the console
7. Save all data to a CSV file

## How This Scraper Was Built — The Full Development Story

This section documents the full journey of building this scraper from scratch. It went through three complete strategy changes and several debugging sessions before landing on the final working approach. Every decision is explained here in the order it happened.

---

### Attempt 1 — requests + BeautifulSoup (Did Not Work)

The first instinct was the standard Python scraping approach: use `requests` to download the page HTML, then parse it with `BeautifulSoup` to find the match data.

```python
import requests
from bs4 import BeautifulSoup

html = requests.get("https://op.gg/lol/summoners/euw/Afroyobro2-EUW").text
soup = BeautifulSoup(html, "html.parser")
```

This works perfectly on simple static websites. OP.GG is not a static website.

When you fetch the OP.GG URL with a plain HTTP request, you get back a nearly empty HTML skeleton. There are no match cards, no KDA values, no rank information — just a shell. This is because OP.GG is built with **Next.js**, a React framework where all the actual content is loaded and rendered by JavaScript running in the browser after the page first loads. `BeautifulSoup` only sees what was in the initial HTTP response, which is essentially nothing useful.

`requests + BeautifulSoup` was abandoned immediately.

---

### Attempt 2 — RSC Payload Interception (Did Not Work)

Modern Next.js applications use something called **React Server Components (RSC)**. Instead of sending full HTML, the server can stream structured component data directly to the client. If you send an HTTP request with the header `RSC: 1`, Next.js skips the HTML shell and returns a raw data payload instead.

The theory was: intercept this stream directly with a `requests` call, parse the structured JSON-like data, and extract match information without needing a browser at all.

The response came back — but with only 307 characters:

```json
{"b":"1776271853","f":[["children",["locale","en","d"],...]],"S":false}
```

This was just the router navigation tree, not summoner data. OP.GG's RSC implementation requires a dynamic internal state token that changes per session and is generated by the browser-side JavaScript. There was no reliable way to replicate it from a plain HTTP request.

RSC interception was abandoned.

---

### Attempt 3 — Playwright Browser Automation

Since the data only renders after JavaScript executes inside a real browser, the solution was to use a real browser — controlled by code. **Playwright** is a Python library that launches a full Chromium browser, navigates pages, clicks buttons, scrolls, and reads the rendered DOM — all programmatically.

#### Installation Issue on Windows

The first problem was installing Playwright. It depends on a package called `greenlet`, which on Windows tried to compile from C++ source code and failed because the installed Visual Studio version was too old:

```
error C7555: use of designated initializers requires at least '/std:c++latest'
```

Fix — force pip to use a pre-compiled binary wheel instead of building from source:

```bash
pip install greenlet --only-binary=:all:
```

#### Bot Detection

With Playwright installed, the browser launched and loaded OP.GG — but the match cards were not rendering. The page loaded but stayed empty where data should be.

OP.GG detects automated browsers using signals like `navigator.webdriver` being set to `true` — a flag that Playwright sets by default. This was patched by injecting a script before any page code ran:

```javascript
Object.defineProperty(navigator, "webdriver", { get: () => undefined });
```

Combined with a real Chrome `User-Agent` string and proper browser context headers, the match cards started appearing correctly.

#### Finding the Right DOM Selectors

With the page rendering, the next step was identifying where the data lived in the HTML. OP.GG uses **Tailwind CSS** — a utility-first CSS framework where every element's styling comes from generic class names like `flex`, `gap-2`, `text-gray-500` rather than descriptive semantic names like `match-card` or `kda-value`. There are no clean class names to target.

A DOM inspector script was written to walk every text node in the page and find which elements contained the text "Victory" or "Defeat", then print their parent hierarchy:

```
Level 0: <STRONG> class='text-[var(--game-item-color-600)] md:text-gray-500'
Level 1: <DIV>   class='flex min-h-[17px] items-center gap-1'
Level 2: <BUTTON> class='flex items-center gap-2 cursor-default'
Level 3: <DIV>   class='flex gap-2 md:flex-col md:gap-0'
```

The match card container was identified as `div.flex.gap-2.md:flex-col.md:gap-0`.

#### KDA Was in Separate Elements

Initially KDA came back as `?/?/?` for every match. The assumption was that kills, deaths, and assists were in a single text node like `5 / 3 / 11`. They were not.

A deeper debug showed the full `innerText` of a match row contained only:

```
Defeat
42m 52s
```

The KDA numbers were in a completely separate sibling `div` — not inside the same container as the result and duration. The fix was to walk UP the DOM tree from the "Victory"/"Defeat" text node, level by level, until reaching an ancestor element whose `innerText` contained BOTH a KDA pattern (`\d+ / \d+ / \d+`) AND a duration pattern (`\d+m \d+s`). That ancestor was the true match card container:

```javascript
let el = node.parentElement;
for (let i = 0; i < 15; i++) {
  const kdaMatch = el.innerText.match(/(\d+)\s*\/\s*(\d+)\s*\/\s*(\d+)/);
  const durMatch = el.innerText.match(/(\d+)m\s*(\d+)s/);
  if (kdaMatch && durMatch) {
    /* extract here */ break;
  }
  el = el.parentElement;
}
```

#### Loading All Matches — The "Show More" Button

By default OP.GG only shows 20 matches. A "Show more" button at the bottom loads additional batches. The script needed to keep clicking it until it disappeared.

First attempt used Playwright's built-in `.click()` method, which failed with a timeout because the button existed in the DOM but was not considered "visible" by Playwright's standards:

```
TimeoutError: ElementHandle.click: Timeout 30000ms exceeded.
element is not visible
```

The fix was to bypass Playwright's visibility checks entirely and fire a raw JavaScript click directly on the DOM element:

```javascript
const btn = Array.from(document.querySelectorAll("button")).find(
  (b) => b.innerText.trim() === "Show more",
);
if (btn) {
  btn.click();
  return true;
}
```

There was also a case-sensitivity issue — the button text is `"Show more"` with a lowercase `m`, not `"Show More"`. A mismatch here caused the script to click the wrong button (`"Show More Detail Games"`) 20 times in a row, expanding individual match detail panels instead of loading new matches.

#### Deduplication

After the DOM tree walk, some matches appeared multiple times because the same text existed in nested parent elements — each level of the hierarchy matched the extraction condition. A JavaScript `Set` was used to deduplicate by building a unique key from result + kills + deaths + assists + duration:

```javascript
const key = `${text}-${kills}-${deaths}-${assists}-${minutes}${seconds}`;
if (!seen.has(key)) { seen.add(key); results.push({...}); }
```

#### Performance Optimizations

The final version runs in headless mode (no visible browser window) and blocks all images, fonts, stylesheets, and media files from loading since none of that is needed for text extraction:

```python
def block_unnecessary(route):
    if route.request.resource_type in ["image", "font", "media", "stylesheet"]:
        route.abort()
    else:
        route.continue_()
```

This reduced page load time significantly since OP.GG loads a large number of champion and item images.

### PUUID Extraction

In parallel with the match scraping, a separate lightweight method was used to extract the summoner's **PUUID** (the unique player identifier used by Riot's internal systems). The PUUID is embedded in the raw HTML of the page as an escaped JSON string, visible even without JavaScript execution:

```python
html = requests.get(SUMMONER_URL, headers=headers).text
puuid_match = re.search(r'\\"puuid\\":\\"([A-Za-z0-9_\-]{60,90})\\"', html)
```

This runs before Playwright launches, so the PUUID is available immediately and gets written into every row of the CSV output.
